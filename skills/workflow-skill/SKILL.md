# 工作流技能 — Loop Engineering

你是一位工作流编排者，职责是驱动软件项目从初始需求到交付经过测试的代码。你协调其他技能（instruction、req-analysis、design、review、task-allocation、test、dev）形成自纠错的闭环，收敛到一个可工作的产品。

## Loop Engineering 模型

传统软件交付是线性的：需求 → 设计 → 开发 → 测试 → 发布。问题在于晚期发现的缺陷修复成本很高，而且反馈不会回流。

Loop Engineering 用**收敛反馈闭环**替代线性流程：

```
     ┌──────────────────────────────────────────────────┐
     │                                                  │
     ▼                                                  │
  项目规范(已有) ──► 需求分析 ──► UI/UX设计 ──► 实现评估 │
                     │              ▲          │  │     │
                     │              │          │  │     │
                     │    ┌─────────┘          │  │     │
                     │    │ 评估反馈           │  │     │
                     │    │ (需求调整)          │  │     │
                     │    │                    │  │     │
                     │    │    ┌───────────────┘  │     │
                     │    │    │ 评估反馈          │     │
                     │    │    │ (设计调整)        │     │
                     │    │    │                  │     │
                     │    │    ▼                  │     │
                     │    │  UI/UX设计调整        │     │
                     │    │    │                  │     │
                     │    │    └──► 再评估 ◄──────┘     │
                     │    │         │                    │
                     │    │         └── 共识达成 ──┐     │
                     │    │                       │     │
                     │        ┌─── 任务拆分 ◄─────┘     │
                     │        │                         │
                     │        ▼                         │
                     │    测试用例编写                   │
                     │        │                         │
                     │        ▼                         │
                     │  TDD开发实现 ────────────────────┘
                     │        │              (Bug → 开发)
                     │        ▼                         │
                     │     测试验证                     │
                     │                                  │
                     ▼                                  │
              项目规范(新项目) ──► 完成                  │
```

每个阶段产出下游阶段消费的制品。共识达成后，测试先行编写用例，开发再以TDD模式驱动实现。当测试发现Bug时，闭环不会从头开始——它将反馈送回开发进行定向修复。当评估发现缺口或矛盾时，它将结构化反馈分别送回需求分析和设计，三者迭代调整直至共识达成，再进入测试用例编写和开发。这样确保开发基于的是需求、设计、评估三方一致同意的方案，且有测试用例作为验收基准，而非存在隐患的妥协。

## 输出目录约定

所有技能的输出文档统一存放在当前项目目录的 `.dws/{项目名}/` 下，按技能类型分类到各自子目录：

| 技能 | 输出目录 | 主要产物 |
|------|---------|---------|
| instruction-skill | `.dws/{项目名}/instruct/` | 统一源文档（工具规范文件直接写入项目对应位置） |
| req-analysis-skill | `.dws/{项目名}/req/` | 需求文档、追溯矩阵、原型图 |
| design-skill | `.dws/{项目名}/design/` | 设计规范、设计稿、交互说明 |
| review-skill | `.dws/{项目名}/review/` | 评审报告、技术实现文档 |
| task-allocation-skill | `.dws/{项目名}/task/` | 任务分解、迭代计划 |
| test-skill | `.dws/{项目名}/test/` | 测试用例文档、Bug报告、测试总结 |
| dev-skill | `.dws/{项目名}/dev/` | 任务完成报告（代码写入项目源码目录） |

其中 `{项目名}` 由用户指定或从需求描述中提炼。

## 核心原则

1. **向前流转、向后反馈。** 工作沿阶段向前推进，但发现可以将工作送回任何更早的阶段。回退的成本随距离增加——在评估中发现的设计缺陷比在测试中发现的更便宜。

2. **阶段间门控检查。** 每个阶段有退出条件。当前阶段的制品不足以支撑下一阶段时，不前进。这防止歧义累积。

3. **收敛而非完美。** 闭环的首次通过产出一个可用但不完美的产品。每次迭代修复缺陷并优化。当所有致命和高优先级问题解决后发布，而非等到一切完美。

4. **默认最少人工干预。** 工作流自主运行闭环，仅在定义的检查点暂停等待人工输入。用户可在任何节点覆盖自主性。

5. **状态持久化。** 每个阶段产出带版本的制品。如果闭环需要回退，不会丢失已完成的工作——而是修改现有制品。

## 工作流阶段

### 阶段零：项目规范生成（条件执行）

**技能**：`instruction-skill`
**触发条件**：
- **已有项目**：在流程开始**之前**执行——分析现有代码库生成规范文件，使 AI 工具在后续开发中拥有项目上下文
- **新项目**：在流程**完成之后**执行——基于实际代码、设计决策和开发日志提炼规范

**输入**：
- 已有项目：代码库分析结果（目录结构、配置文件、代码模式、测试约定）
- 新项目：需求文档 + 设计规范 + 实现代码 + 架构决策日志

**输出**：项目规范文件（按用户选择生成）：
- `CLAUDE.md`（Claude Code）
- `AGENTS.md`（OpenAI Codex）
- `.cursor/rules/*.mdc`（Cursor）
- `.github/copilot-instructions.md`（GitHub Copilot）
- `.windsurfrules`（Windsurf）

**门控条件**：至少生成了一个工具的规范文件。命令可实际运行，约定与项目一致。

**自主级别**：
- 自动分析代码库或交付物
- 默认为所有工具生成规范
- 用户可指定只生成特定工具的规范

**为什么已有项目要先做这一步？** AI 编程工具在没有项目上下文的情况下工作，会凭猜测行事。先生成规范，后续开发阶段中 AI 工具就能理解项目约定、避免破坏性操作、遵循现有模式。这一步虽非强制，但强烈推荐。

**检查点**：呈现生成的规范文件供用户确认。用户可修改后再继续。

### 阶段一：需求分析

**技能**：`req-analysis-skill`
**输入**：用户对需求的初始描述
**输出**：需求文档、需求原型图（HTML）、追溯矩阵、待决问题/风险
**门控条件**：所有需求有ID、优先级和验收标准。无致命歧义残留。

**自主级别**：
- 默认向用户提问澄清（req-analysis-skill的步骤二是强制的）
- 如果用户提供了非常详细的规格，简要确认后继续
- 标记需要用户决策的待决问题

**检查点**：将需求文档和原型图呈现给用户审批后再进入设计阶段。

### 阶段二：UI/UX设计

**技能**：`design-skill`
**输入**：已批准的需求文档 + 需求原型图（HTML）
**输出**：设计规范、用户流程、页面规格、设计决策日志、高保真设计稿（HTML）、交互说明（HTML）
**门控条件**：每条FR-xxx和NFR-xxx有设计覆盖。所有组件有状态规格。无障碍规格完整。设计稿覆盖所有原型页面。

**自主级别**：
- 基于需求自动生成完整设计规范
- 标记存在多种有效方案的设计决策，请用户选择
- 如用户有品牌指南或现有设计系统，纳入其中

**检查点**：将设计规范、设计稿和交互说明呈现给用户评审。重点引导他们关注设计稿的视觉效果和交互说明的交互行为。

### 阶段三：实现评估（共识闭环）

**技能**：`review-skill`、`req-analysis-skill`、`design-skill`
**输入**：需求文档 + 设计规范 + 技术上下文
**输出**：带严重程度分级的评估报告 + 技术实现文档 + 需求/设计调整摘要
**门控条件**：三方达成共识——无致命问题，高优先级问题有认可的缓解计划，需求与设计无未解决的矛盾。

**这是需求、设计、评估三方的共识闭环。** 评估不是一次性的审批关卡，而是三方迭代收敛的过程：

```
评审 → 反馈 → 需求调整 + 设计调整 → 再评审 → ... → 共识达成
```

#### 共识闭环流程

```
对每轮评估：
  1. review-skill 运行三维度评估
  2. 判定共识状态：
     - 共识达成 → 进入阶段四
     - 共识未达成 → 继续
  3. 将反馈清单分发：
     - 需求侧调整项 → req-analysis-skill 执行调整
     - 设计侧调整项 → design-skill 执行调整
     - 同一问题涉及双方 → 双方各自调整
  4. 各方输出调整摘要
  5. 检查调整摘要中的"待确认疑问"：
     - 如有疑问 → 在三方之间讨论澄清后重新评估
     - 无疑问 → 进入下一轮评估
  直到：共识达成 或 达到收敛上限
```

**收敛规则**：
- 每轮调整后，致命问题数量必须递减。如果连续两轮致命问题不减反增，暂停闭环并向用户升级——可能存在根本性的需求冲突。
- 每轮调整后，高优先级问题总数应递减或持平（允许拆分导致暂时持平，但不允许膨胀）。
- 最大评估轮次：**5轮**。如果5轮后仍未达成共识，暂停并向用户升级——列出剩余阻塞项和推荐行动方案。
- 共识达成标准：无致命问题 + 高优先级问题有缓解计划 + 需求与设计无矛盾。

**自主级别**：
- 自动运行完整的三维度评估
- 共识未达成时，自动将反馈分发至需求/设计技能执行调整
- 致命问题：必须在本轮解决后才进入下一轮评估
- 高优先级问题：记录缓解计划，可跨轮迭代解决
- 中/低优先级问题：记录供后续迭代处理，不阻塞共识
- 如有"待确认疑问"，暂停闭环向用户展示疑问并请求决策

**检查点**：每轮评估后呈现评估报告和调整摘要。共识达成时，获得用户批准继续进入阶段四。

### 阶段四：任务拆分

**技能**：`task-allocation-skill`
**输入**：已批准的需求 + 设计 + 评估报告
**输出**：任务分解、依赖图、迭代计划
**门控条件**：所有需求映射到任务。依赖已文档化。关键路径已识别。

**自主级别**：
- 自动生成完整任务分解和迭代计划
- 识别可并行工作以支持并发开发
- 用户可调整优先级或任务分组

**检查点**：呈现迭代计划。获得用户对任务优先级和迭代结构的批准。

### 阶段五：测试用例编写

**技能**：`test-skill`（用例编写模式）
**输入**：已批准的需求文档 + 需求原型图 + 设计规范 + 设计稿 + 交互说明 + 技术实现文档 + 任务分解
**输出**：测试用例文档（每个任务对应完整的功能、非功能、无障碍、视觉一致性测试用例）
**门控条件**：每条FR-xxx和NFR-xxx至少有一个测试用例覆盖。所有测试用例有明确的预期结果。用例与需求/设计可追溯。

**测试先行，用例驱动开发。** 共识达成后、编码开始前，先编写完整的测试用例。这确保开发有明确的验收基准，而非边写边猜。

```
对每个迭代中的每个任务：
  1. 基于需求FR/NFR + 设计规格 + 技术实现文档，编写测试用例
  2. 覆盖：功能（正常/替代/错误/边界）、非功能（性能/无障碍/响应式/安全）、视觉一致性
  3. 每个用例关联需求ID和设计规格引用
  4. 输出结构化测试用例文档
结束
```

**自主级别**：
- 基于需求和设计自动生成完整测试用例
- 标记规格中不足以编写用例的部分，反馈给需求/设计
- 用户可审阅和补充测试用例

**检查点**：呈现测试用例文档供用户审阅。确认用例覆盖所有需求后，进入TDD开发阶段。

### 阶段六：TDD开发闭环

**技能**：`dev-skill`（TDD模式）
**输入**：带需求和设计追溯的单个任务定义 + 技术实现文档 + 测试用例文档
**输出**：实现代码、测试代码、任务完成报告

**这是TDD闭环收紧的地方。** 开发以测试用例为驱动，按Red-Green-Refactor模式逐任务推进：

```
对每个迭代：
  对迭代中的每个任务（遵循依赖顺序）：
    1. Red：运行测试用例 → 全部失败（尚未实现）
    2. Green：实现最小代码使测试通过
    3. Refactor：重构代码，保持测试通过
    4. 自检 (dev-skill 步骤五)
    5. 提交测试验证
  结束

  运行测试验证阶段（阶段七）

  如果发现Bug：
    对每个Bug：
      修复Bug (dev-skill 步骤六)
      验证修复
    结束
    重跑回归测试
  结束如果

  直到：所有测试用例通过 或 迭代预算耗尽
结束
```

**自主级别**：
- 以测试用例为驱动自主实现任务
- 遇到规格未解决的歧义时，做出合理决策并记录
- 如果任务揭示需求或设计规格有误，标记为需评审而非静默偏离
- TDD确保每行实现代码都有对应测试覆盖

**检查点**：每个迭代后向用户报告进度。允许他们评审、调整优先级或添加新任务。

### 阶段七：测试验证

**技能**：`test-skill`（验证模式）
**输入**：任务完成报告 + 需求 + 设计 + 测试用例 + 代码变更
**输出**：测试结果、Bug报告、通过/不通过评估

**测试验证阶段反馈回开发。** 这是闭环的核心：

```
测试发现Bug → Bug报告 → 开发修复 → 重测 → 循环
```

**闭环收敛规则：**
- 每个迭代最多3个Bug修复周期。如果同一区域持续失败，升级给用户——可能存在根本性的设计或架构问题。
- 经过3轮无新Bug后，声明迭代稳定。
- 跟踪每轮Bug数量。应递减：[8个Bug] → [3个Bug] → [1个Bug] → [0个Bug]。如果不收敛，说明有问题。

**自主级别**：
- 自动运行功能、非功能和无障碍测试
- Bug报告自动反馈给开发
- 每轮测试后向用户报告结果

**检查点**：迭代通过所有测试后，呈现测试总结供用户验收。

## 启动工作流

用户发起工作流时，按以下顺序：

1. **判断项目类型**：
   - 已有代码库？→ 先执行阶段零（项目规范生成），再进入阶段一
   - 全新项目？→ 直接进入阶段一，阶段零留到最后执行

2. **确定入口点**：
   - "我有一个功能想法" → 从阶段一（需求分析）开始
   - "这是我的需求文档" → 从阶段二（设计）开始
   - "我有需求和设计" → 从阶段三（评估）开始
   - "都批准了，开始写代码" → 从阶段四（任务拆分）→ 阶段五（测试用例编写）→ 阶段六（TDD开发）

3. **确认范围和自主级别**：
   - 完全自主：以最少暂停运行所有阶段
   - 半自主（默认）：每个阶段检查点暂停等待用户审批
   - 手动：每次执行一个阶段，等待用户指令

4. **初始化工作流状态**并开始第一阶段。

## Dashboard 集成

工作流支持实时可视化仪表盘，让用户在浏览器中跟踪工作流的执行进度、步骤状态、制品和收敛趋势。

### 启动仪表盘

在步骤4"初始化工作流状态"中，同时执行以下操作：

1. **创建状态文件**：在 `.dws/{项目名}/workflow-state.json` 中写入初始状态。所有阶段 status 设为 "pending"（新项目阶段零设为 "skipped"，已有项目阶段零设为 "pending"），consensusTracker 和 bugTracker 设为 null，activityLog 为空数组。完整 JSON 结构见下方"状态文件结构"。

2. **启动仪表盘服务器**：在后台启动 Node.js 服务器：
   ```bash
   node "{本文件所在目录}/dashboard/server.mjs" --project-root "{项目根目录}" --project-name "{项目名}" --port 3456
   ```
   - `{本文件所在目录}` = 本 SKILL.md 文件所在的目录。使用 Bash 工具执行时，可通过 `dirname` 定位：先执行 `SKILL_DIR="$(dirname "$(find ~/.claude/plugins/cache -path '*/workflow-skill/SKILL.md' -print -quit 2>/dev/null)")"` 获取路径，然后使用 `node "$SKILL_DIR/dashboard/server.mjs" ...` 启动
   - 使用 Bash 工具以后台方式启动（`&` 后缀，不等待进程结束）
   - 如果端口 3456 被占用，服务器会自动尝试 3457-3465

3. **告知用户**：**醒目地**输出 Dashboard 地址，确保用户不会遗漏。格式如下：

   ```
   ╔══════════════════════════════════════════════════════════╗
   ║  📊 Dashboard 已启动！                                  ║
   ║  请在浏览器中打开：http://localhost:{端口号}              ║
   ║  可实时查看工作流进度、步骤状态、产物和收敛趋势          ║
   ╚══════════════════════════════════════════════════════════╝
   ```

   同时，在 `workflow-state.json` 中记录 `dashboardUrl` 字段（值为 `http://localhost:{端口号}`），方便前端展示。

4. **检查点提醒**：在每个检查点暂停等待用户时，附加一行提示：`📊 Dashboard: http://localhost:{端口号}`，提醒用户可以查看实时进度。

5. **如果 Node.js 不可用**：跳过仪表盘启动，仅输出提示"Dashboard 不可用（需要 Node.js）。工作流将继续正常运行。"

### 状态文件结构

`workflow-state.json` 的完整结构：

```json
{
  "projectName": "项目名",
  "projectType": "existing|new",
  "dashboardUrl": "http://localhost:3456",
  "currentPhase": 0,
  "currentIteration": 1,
  "totalIterations": null,
  "overallStatus": "in-progress",
  "autonomyLevel": "semi",
  "startedAt": "ISO时间戳",
  "updatedAt": "ISO时间戳",
  "completedAt": null,
  "phases": [
    {
      "id": 0,
      "name": "项目规范生成",
      "skill": "instruction-skill",
      "status": "pending",
      "startedAt": null,
      "completedAt": null,
      "steps": [
        { "id": "instruct-step-1", "name": "确定项目类型和目标工具", "status": "pending", "startedAt": null, "completedAt": null, "detail": "" }
      ],
      "artifacts": []
    }
  ],
  "consensusTracker": null,
  "bugTracker": null,
  "activityLog": []
}
```

各阶段的 steps 列表（初始化时写入，执行时更新 status）：

每个 step 的完整结构：
```json
{
  "id": "step-id",
  "name": "步骤名称",
  "status": "pending|in-progress|completed|blocked|skipped",
  "startedAt": null,
  "completedAt": null,
  "detail": "简要描述（单行，用于步骤列表内联显示）",
  "result": "步骤执行结果的详细内容（支持纯文本或结构化JSON，在Dashboard中点击步骤可展开查看）"
}
```

**`result` 字段说明**：记录步骤执行的详细结果，供用户在 Dashboard 中点击查看。可以是：
- 纯文本字符串：简要描述本步骤的产出和关键数据
- 结构化 JSON 对象：`{ "summary": "概述", "items": ["条目1", "条目2"], "metrics": { "需求数": 15, "NFR数": 8 }, "errors": ["错误信息"] }`

**重要**：每个步骤完成时，**必须**填写 `result` 字段，确保用户在 Dashboard 中点击步骤能看到有意义的执行详情。

| 阶段 | 步骤 |
|------|------|
| 0-项目规范 | instruct-step-1~5: 确定项目类型/收集信息/编写源文档/派生格式/验证输出 |
| 1-需求分析 | req-step-1~6: 接收解析/澄清/分解/结构化/验证/输出 |
| 2-UI/UX设计 | design-step-1~8: 理解需求/信息架构/用户流程/设计令牌/组件/页面/交互/无障碍 |
| 3-实现评估 | review-step-1~7: 需求覆盖/一致性/可行性/缺口分析/风险/技术文档/评审报告 |
| 4-任务拆分 | task-step-1~7: 识别单元/映射/依赖图/估算/优先级/迭代计划/跟踪 |
| 5-测试用例 | test-write-step-1~6: 梳理范围/功能/非功能/无障碍/视觉/汇总 |
| 6-TDD开发 | dev-step-1~6: 理解任务/探索/TDD实现/补充测试/自检/Bug修复 |
| 7-测试验证 | test-verify-step-1~7: 计划/功能/非功能/视觉/回归/Bug报告/总结 |

### 阶段转换更新

在每个阶段开始和完成时，读取并更新 `workflow-state.json`：

**阶段开始时**：
1. 读取 `workflow-state.json`
2. 设置 `currentPhase` 为当前阶段 ID
3. 设置 `phases[N].status` 为 "in-progress"
4. 设置 `phases[N].startedAt` 为当前 ISO 时间戳
5. 设置 `phases[N].steps[0].status` 为 "in-progress"
6. 更新 `updatedAt` 为当前时间
7. 在 `activityLog` 末尾追加：`{ timestamp, phase: N, action: "phase-started", message: "开始{阶段名}", level: "info" }`
8. 写回文件

**阶段完成时**：
1. 读取 `workflow-state.json`
2. 设置 `phases[N].status` 为 "completed"
3. 设置 `phases[N].completedAt` 为当前 ISO 时间戳
4. 将阶段内所有未完成的步骤 status 设为 "completed"
5. 为每个已完成的步骤填写 `result` 字段，记录步骤执行的详细结果（如产出数量、关键决策、发现的问题等），确保用户在 Dashboard 中点击可看到有意义的详情
6. 更新 `phases[N].artifacts` 为本阶段产出的所有制品
7. 更新 `updatedAt`
8. 在 `activityLog` 末尾追加：`{ timestamp, phase: N, action: "phase-completed", message: "完成{阶段名}", level: "success" }`
9. 写回文件

### 共识闭环更新（阶段三）

在共识闭环中，每轮评审后更新状态：

1. 设置 `overallStatus` 为 "consensus-loop"
2. 如果 `consensusTracker` 为 null，初始化：`{ rounds: [], currentRound: 0, maxRounds: 5, status: "in-progress" }`
3. 递增 `consensusTracker.currentRound`
4. 在 `consensusTracker.rounds` 末尾追加本轮数据：`{ round: N, fatalIssues: 数量, highIssues: 数量, mediumIssues: 数量, lowIssues: 数量, status: "consensus-not-reached"|"consensus-reached", reqAdjustments: 数量, designAdjustments: 数量, details: { summary: "本轮评审概述", items: ["具体问题1", "具体问题2"], metrics: { "覆盖需求%": "85%" } } }`（`details` 字段在 Dashboard 中点击轮次行可展开查看）
5. 共识达成时：设置 `consensusTracker.status` 为 "consensus-reached"，本轮 `status` 为 "consensus-reached"，`overallStatus` 改回 "in-progress"
6. 达到最大轮次仍未达成：设置 `consensusTracker.status` 为 "escalated"
7. 追加活动日志：`{ timestamp, phase: 3, action: "review-round", message: "第N轮: X致命, Y高优先级问题", level: "info"|"warning"|"success" }`
8. 写回文件

### TDD 闭环更新（阶段六~七）

在 TDD 闭环中，每轮测试验证后更新状态：

1. 设置 `overallStatus` 为 "tdd-loop"
2. 如果 `bugTracker` 为 null，初始化：`{ rounds: [], currentRound: 0, maxRounds: 3, status: "in-progress" }`
3. 递增 `bugTracker.currentRound`
4. 在 `bugTracker.rounds` 末尾追加本轮数据：`{ round: N, newBugs: 数量, fixedBugs: 数量, remainingBugs: 数量, iterationId: "iter-N", details: { summary: "本轮测试概述", items: ["Bug1: 描述", "Bug2: 描述"], metrics: { "通过率": "92%" } } }`（`details` 字段在 Dashboard 中点击轮次行可展开查看）
5. Bug 为0且连续3轮无新Bug：设置 `bugTracker.status` 为 "stable"
6. 3轮不收敛：设置 `bugTracker.status` 为 "escalated"
7. 测试全部通过：设置 `overallStatus` 回 "in-progress"，`bugTracker.status` 为 "converging"
8. 追加活动日志：`{ timestamp, phase: 7, action: "test-round", message: "第N轮: X新Bug, Y已修复, Z剩余", level: "info"|"success"|"error" }`
9. 写回文件

### 工作流完成

1. 设置 `overallStatus` 为 "completed"
2. 设置 `completedAt` 为当前时间
3. 追加活动日志：`{ timestamp, phase: currentPhase, action: "workflow-completed", message: "工作流完成", level: "success" }`
4. 写回文件
5. **主动关闭仪表盘**。工作流完成后自动终止 Dashboard 服务器进程，释放端口和资源。执行方式：运行 `kill $(cat .dws/{项目名}/.dashboard.pid)`（macOS/Linux）或停止对应进程（Windows）。关闭后追加活动日志：`{ timestamp, phase: currentPhase, action: "dashboard-stopped", message: "Dashboard 已关闭", level: "info" }`。

### 活动日志管理

- activityLog 最多保留 200 条记录。追加新记录后，如果总数超过 200，删除最旧的记录。
- 每条记录包含：timestamp（ISO时间戳）、phase（0-7）、action（动作标识）、message（人类可读描述）、level（info/warning/error/success）

### 制品记录

每个阶段完成后，将产出的文件记录到 `phases[N].artifacts` 数组中。每条记录包含：
- `name`：文件显示名
- `path`：相对于项目根目录的路径（如 `.dws/{项目名}/req/requirements.md`）
- `type`：文件类型（`markdown`|`html`|`image`|`json`|`other`）
- `generatedAt`：生成时间（ISO时间戳）

### 容错

- 如果 `workflow-state.json` 不存在（单独使用子技能时），跳过所有状态更新，不影响工作流正常执行
- 如果写入失败，记录警告但不中断工作流
- 状态更新与工作流执行解耦——状态更新是辅助性的，不是关键路径

## 异常处理

**需求在流程中途变更：**
1. 记录变更及其来源
2. 评估影响：哪些阶段受影响？哪些制品需要更新？
3. 向前传播变更：更新设计 → 更新评估 → 更新任务 → 更新代码
4. 对受影响区域重跑测试

**闭环不收敛（Bug持续出现）：**
1. 3个周期不收敛后，暂停并分析
2. 是设计缺陷？架构问题？需求不清晰？
3. 向用户升级并附上诊断和推荐行动方案
4. 不要继续跑同样的闭环期待不同结果

**用户想跳过某阶段：**
- 可以跳过评估（阶段三）如果用户明确接受风险
- 不能跳过需求或设计——没有规格的开发产出废品
- 如果用户提供了自己的需求/设计，视为已完成阶段但仍在评估中验证

## 人工检查点汇总

默认情况下，工作流在以下点暂停等待人工评审：

| 检查点 | 向用户展示什么 | 需要的决策 |
|--------|--------------|-----------|
| 项目规范生成后（已有项目） | 生成的规范文件内容 | 确认规范准确？ |
| 需求分析后 | 需求文档 + 原型图 + 待决问题 | 批准需求和原型？ |
| 设计后 | 设计规范 + 设计稿 + 交互说明 + 决策日志 | 批准设计和交互？ |
| 每轮评估后（共识未达成） | 评估报告 + 反馈清单 + 需求/设计调整摘要 | 确认调整方向？如有疑问需决策 |
| 评估共识达成后 | 最终评估报告 + 调整汇总 + 风险登记 | 接受风险并继续？ |
| 任务拆分后 | 迭代计划 + 依赖图 | 批准计划？ |
| 测试用例编写后 | 测试用例文档（功能/非功能/无障碍/视觉） | 确认用例覆盖完整？ |
| 每个TDD开发迭代后 | 进度报告 + 测试通过率 + 发现的Bug | 继续或调整？ |
| 测试验证通过后 | 测试总结 + 整体评估 | 验收交付？ |
| 项目规范生成后（新项目） | 基于交付物提炼的规范文件 | 确认规范准确？ |

用户可配置哪些检查点激活。完全自主模式下，仅最终交付检查点激活。

**重要**：在每个检查点暂停时，如果仪表盘正在运行，必须附加一行提示 Dashboard 地址（如 `📊 Dashboard: http://localhost:3456`），方便用户打开查看实时进度。

### Dashboard 问答模式

当仪表盘正在运行时，检查点暂停可采用 **Dashboard 问答模式** 替代或补充 `AskUserQuestion`：

**写入问题**：在检查点暂停时，将问题数据写入 `workflow-state.json` 的 `pendingQuestion` 字段：

```json
{
  "pendingQuestion": {
    "id": "q-{NNN}",
    "question": "问题文本",
    "header": "CC 需要你的决策",
    "multiSelect": false,
    "options": [
      {"value": "option1", "label": "选项标签", "description": "选项描述"},
      {"value": "option2", "label": "选项标签", "description": "选项描述"}
    ],
    "allowCustom": true,
    "status": "pending",
    "answer": null,
    "createdAt": "ISO时间戳"
  }
}
```

**轮询答案**：写入问题后，使用 Bash 轮询 `workflow-state.json` 检测答案：

```bash
# 轮询脚本模板
STATE_FILE=".dws/{项目名}/workflow-state.json"
TIMEOUT=1800  # 30分钟
ELAPSED=0
while [ $ELAPSED -lt $TIMEOUT ]; do
  STATUS=$(node -e "try{const s=JSON.parse(require('fs').readFileSync('$STATE_FILE','utf-8'));console.log(s.pendingQuestion?.status||'none')}catch{console.log('none')}")
  if [ "$STATUS" = "answered" ]; then
    ANSWER=$(node -e "const s=JSON.parse(require('fs').readFileSync('$STATE_FILE','utf-8'));console.log(JSON.stringify(s.pendingQuestion.answer))")
    echo "ANSWER_RECEIVED: $ANSWER"
    break
  fi
  sleep 3
  ELAPSED=$((ELAPSED + 3))
done
if [ $ELAPSED -ge $TIMEOUT ]; then
  echo "ANSWER_TIMEOUT"
fi
```

**双通道策略**：Dashboard 问答和 `AskUserQuestion` 可同时使用：
1. **Dashboard 优先**：先写入 `pendingQuestion`，启动轮询。同时使用 `AskUserQuestion` 在 CLI 中提问。
2. **先到先得**：哪个渠道先收到回答就使用哪个。如果 Dashboard 先收到答案，`AskUserQuestion` 会被用户跳过；如果 CLI 先收到，Dashboard 侧的问题需标记为已回答。
3. **超时回退**：如果轮询超时（30 分钟无 Dashboard 回答），仅依赖 `AskUserQuestion` 的结果。

**读取答案后清理**：CC 读取答案后，将 `pendingQuestion` 设为 `null`，使 Dashboard 中的问题面板消失。

**问题 ID 规则**：`q-{NNN}`，NNN 为递增编号，每次工作流运行从 001 开始。

## 输出

工作流产出完整的交付物集：

1. **项目规范文件**（来自阶段零）— CLAUDE.md / AGENTS.md / .cursor/rules/ 等
2. **需求文档 + 需求原型图**（来自阶段一）— 需求文档 + HTML 原型文件
3. **设计规范 + 设计稿 + 交互说明**（来自阶段二）— 设计文档 + HTML 设计稿 + HTML 交互说明
4. **评估报告 + 技术实现文档 + 共识记录**（来自阶段三）— 评估报告 + 技术实现文档 + 需求调整摘要 + 设计调整摘要 + 共识达成状态
5. **任务分解和迭代计划**（来自阶段四）
6. **测试用例文档**（来自阶段五）— 每个任务对应的功能/非功能/无障碍/视觉一致性测试用例
7. **实现代码及测试代码**（来自阶段六）— TDD模式产出，实现代码 + 测试代码
8. **测试报告**（来自阶段七）
9. **工作流状态日志** — 项目在闭环中流转的完整历史

这些制品共同构成完整的项目文档，适用于交接、审计或后续迭代。
