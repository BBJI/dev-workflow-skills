# 需求质量检查清单

在步骤五（验证）中使用此清单，确保每条需求满足质量标准。

## 单条需求质量标准（INVEST+T）

每条需求应满足：

| 标准 | 检查问题 | 通过？ |
|------|---------|--------|
| **独立** | 此需求能否独立实现和测试（除显式标注的依赖外）？ | |
| **可协商** | 实现方式是否有讨论空间？（不规定怎么做） | |
| **有价值** | 是否对某个干系人交付了明确的价值？ | |
| **可估算** | 开发者能否给出粗略的工作量估算？ | |
| **小型** | 是否小到可以在一个迭代/冲刺内完成？ | |
| **可测试** | 是否可以通过测试客观验证？ | |
| **可追溯** | 是否关联到来源和下游产物？ | |

## 完整性检查

- [ ] 所有用户角色已识别，权限已定义
- [ ] 每个实体都有 CRUD 操作说明（创建、读取、更新、删除）
- [ ] 每个用户操作都有成功路径和失败路径
- [ ] 空状态已文档化（无数据时显示什么？）
- [ ] 加载状态已文档化
- [ ] 错误状态和错误消息已指定
- [ ] 并发访问/竞态条件已处理
- [ ] 数据验证规则已指定（格式、范围、必填/选填）
- [ ] 数据生命周期：创建、保留、归档、删除
- [ ] 每个操作的认证和授权规则
- [ ] 敏感操作的审计/日志需求

## 一致性检查

- [ ] 没有互相矛盾的需求
- [ ] 术语使用一致（对照术语表）
- [ ] 优先级内部一致（没有"Must"依赖于"Could"的情况）
- [ ] 没有重复需求（相同意图，不同措辞）

## 非功能完整性

- [ ] 性能指标已指定（响应时间、吞吐量、并发数）
- [ ] 安全需求已识别（认证、加密、数据保护）
- [ ] 无障碍合规级别已声明（WCAG 2.2 AA等）
- [ ] 可靠性/可用性目标已定义（可用率、RTO、RPO）
- [ ] 可扩展性预期已文档化
- [ ] 浏览器/设备/操作系统兼容性已指定
- [ ] 国际化/本地化需求已识别

## 模糊性红旗

留意这些词语——它们几乎总是表明需求模糊：
- "快速"、"慢"、"迅速"、"响应快"
- "好用"、"直观"、"简单"、"易用"
- "可扩展"、"灵活"、"健壮"、"高效"
- "安全"、"可靠"
- "适当"、"合理"、"足够"
- "等等"、"之类的"、"以及其他"

将每个替换为可测量、可测试的标准。
